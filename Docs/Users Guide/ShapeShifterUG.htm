1           Introduction

1.1         What is ShapeShifter?

ShapeShifter is a framework for rapidly creating interactive 2D and 3D graphics applications. It simplifies Microsoft’s DirectX 8 API by providing a powerful and easy to use retained mode graphics interface. The ShapeShifter Component Library (SSCL) consists of language-independent COM components written in C++ that can be used in with any language that supports COM such as Visual C++, Visual Basic, VBScript, JavaScript, and Python. With ShapeShifter, software developers and graphics artists can quickly add interactive 2D and 3D graphics to applications and websites using the language of their choice.

ShapeShifter is designed for ease of use, flexibility, and high performance. It can be easily customized and used for a wide range of applications.

·        CAD/CAM

·        Simulation

·        Architectural, engineering, and construction

·        Commercial interior design

·        Electrical circuit and computer chip design

·        Interactive product catalogs

·        Computer and telecommunications network visualization

·        Molecular modeling

·        Data visualization

·        Collaborative prototyping

ShapeShifter lowers the cost of entry into 3D graphics and provides a productive framework for building interactive graphics applications.

2          Getting Started

2.1        Installing DirectX 8

ShapeShifter requires DirectX 8. You can install either the DirectX 8 SDK or just the DirectX 8 run-time. See readme.txt for instructions on where to download DirectX. If you install the debug version of the DirectX SDK, be aware that debug tracing for DirectX can be controller from the control panel. The higher you set the debug trace level in the control panel, the slower DirectX will run. If you have performance problems with ShapeShifter, check this setting in the control panel.

2.2        Using ShapeShifter in Visual Basic

From the main menu, select Project|Components.  Scroll down the list and put a check next to ShapeShifter 1.0 Type Library and then click OK. The ShapeShifter control will be added to your tool palette.

 

 

Next, click on the ShapeShifter icon in the tool palette and drop a control onto a form.

 

 

You can now access the properties of the control at either design-time or run-time. Some properties are only available at run-time.

 

 

Notice the ControllerType, ModelType, and ViewportType properties in the property browser. These properties provide a high level of customization to developers by allowing custom models, viewports, and controllers to be plugged into the control. The value of these properties is the ProgId association with the COM class that implements the object.

You can also use ShapeShifter in Visual Basic without using the full ActiveX control. The full control is basically just a wrapper for the rest of the framework that can be inserted into a form or other ActiveX control container. You can declare and use models, viewports, and controllers directly in your application and have the viewports rendered to your form or any other window in your application. The Solar System and Maze samples use this technique.

If you want the ShapeShifter control to load up a model when it is activated, provide a filename to the ModelFile property. You can also leave it blank and the load the model programmatically.

Here is an example of loading a cube into the model programmatically.

 

Private Sub Form_Load()

    Dim Cube As CompositeShape

    Dim MeshEditor As MeshEditor

    Set MeshEditor = New MeshEditor

    Set Cube = ShapeShifterCtl1.Model.CreateNode(

                        "MyCube",

                        "ShapeShifter.CompositeShape",

                        "")

    MeshEditor.BeginEdit Cube.Mesh

    MeshEditor.MakeCube 5, 5, 5

    MeshEditor.EndEdit

End Sub

 

2.3        Using ShapeShifter in Visual C++

The easiest way to use ShapeShifter in Visual C++ is to include the file SSCL_Client.h in your precompiled header. Just add the following line to your stdafx.h file.

#include "SSCL_Client.h"

Make sure that the ShapeShifter include directory is in your Visual C++ include path. Otherwise, you will need to include the full path to SSCL_Client.h in your #include directive.

SSCL_Client.h does a #import on the ShapeShifter type library and generates wrapper classes for the COM objects in the type library.  For more information on #import consult the MSDN library.

The next step is to declare and initialize the ShapeShifter engine in your application. If you are using MFC, then your CWinApp derived class is the best place to declare the engine. You can then initialize it in the InitInstance method.

Next, your application must get a device from the engine object. To get the default device, call IEngine::GetDefaultDevice or you can iterate through the available devices using IEngine::GetDeviceCount and IEngine::GetDevice and choose one that is appropriate for your application. The rendering device is usually an application-wide setting, so you can store it as a member of your CWinApp derived class along with the engine.

Once you have a device, you can create viewports and attach them to the windows in your application. If you are developing an MFC MDI or SDI application, then the viewport typically goes in your CView derived class. The view’s OnInitialUpdate is a good place to create the viewport.

Next, you need to create a model to render in the viewport. In an MFC MDI or SDI application, the model will be declared in your CDocument derived class. The model can be attached to the viewport when the viewport is created.

Finally, you need a controller to provide the user interface to the model and viewport. In an MFC MDI or SDI application, your CView derived class is a good place to do this. It can be created in the view’s OnInitialUpdate and wired to the viewport at that time.

See the BasicShapes MFC sample for an example of how to create and initialize ShapeShifter in Visual C++.

3          Conceptual Overview

ShapeShifter is designed to be modular, easy to use, easily customized, and fast. The classes and interfaces defined by ShapeShifter provide a retained mode interface to DirectX 8. A retained mode interface renders a scene by traversing a hierarchy of objects that can be loaded and saved by the application. In other words, a scene can be rendered from a set of objects that describe the scene. In contrast, immediate mode interfaces like DirectX and OpenGL only provide a set of low-level rendering commands that must be called from a compiled language like C++ or Visual Basic.

3.1        Models, Views, and Controllers

One of the central concepts of ShapeShifter is the model-view-controller (MVC) design pattern. This design pattern provides a clear separation between the data, the view of the data, and the user interface. A model contains the objects (also referred to as nodes) that are rendered in a view. The model is the data that describes a scene. The view (i.e viewport) represents a rectangle on the screen in which a model is rendered. The controller receives input from the user and translates it into actions and commands that are applied to the model and viewport.

The cool thing about using MVC in ShapeShifter is that models, viewports, and controllers can be mixed and matched. This is especially useful when customizing the user interface. Developers can write their own custom controllers and use them instead of the stock controllers provided with the library.

3.1.1          Models

A model is a type of composite node that is rendered by a viewport and manipulated by a controller. A model may contain shapes, textures, lights, materials, and other types of nodes. A model can even contain other models. It contains the retained-mode data that describes a scene. All models implement the IModel interface, which is derived from IVisual. The IModel interface defines methods for manipulating the nodes contained by the model.

3.1.2         Viewports

A viewport is a two-dimensional view of a model. Viewports render a model onto a device. Viewports come in two flavors: windowed and full-screen. Windowed viewports exist inside of a window and are clipped to the window. A full-screen viewport takes up the entire screen and only one can be active at a time.

Viewports can also render additional visual information that does not exist inside the model such as bounding boxes, coordinate axes, and grids. These additional viewport specific objects are referred to as decorators because they provide additional visual aids and window dressing to the viewport, but are not actually part of the model.

3.1.3         Controllers

A controller is an object that receives user input and translates it into actions and commands on the model and viewport. ShapeShifter provides several built-in controller types that provide different functionality. A controller contains one or more behaviors, which are reusable objects that implement a slice of functionality for the controller. Behaviors are reusable because the same behavior can be used in many different controllers.

Controllers can receive input in several ways. They can receive window events from a window, input from DirectInput devices, or animation events from an animation loop. Depending on the controller and the behaviors it exposes, a controller might receive input from one or more these sources.

All controllers implement the IController interface, which is derived from the ICommandDispatcher interface. The ICommandDispatcher interface defines methods for executing and undoing commands and provides the basis for undo/redo functionality. Commands can be sent to the controller through the ICommandDispatch::Do method. A command is an object that encapsulates an action performed on some target object. Command objects are an example of the functor design pattern. To sum it up, controllers are the point of contact for user input and commands that affect the model and viewport.

As mentioned earlier, controllers can receive events from a window. Controllers that do this implement the IWndMessageReceiver interface. This interface allows a controller to be plugged into a message source, which is an object that implements the IWndMessageSource interface. The IWndMessageReceiver interface defines a single method – ProcessWindowMessage. Those of you familiar with ATL will recognize this method and its signature. It takes the message ID, a WPARAM, and an LPARAM and handles a message. By default, controllers forward these messages to any behaviors that are interested in receiving window messages. The behaviors then determine the state change or action that occurs in response to the message.

Controllers can also receive input from DirectInput devices. The IController interface provides methods to attach and detach input devices. Behaviors associated with a controller can acquire the devices they are interested in from the controller. Each input device belongs to a category and the controller can only have one device per category. The categories are defined by the InputDeviceCategory enumeration.

Finally, controllers can receive input from an animation loop. An animation loop fires events at precise intervals. Controllers that do animation implement the IAnimate interface and can be attached to an animation loop. By default, the controller forwards the animation events to behaviors that are listening for those events. Not all controllers implement animation. For example, animation doesn’t make sense for a controller designed to edit a scene. If the controller is designed to allow the user to fly through a scene, then animation makes a lot of sense.

3.2        Nodes and Hierarchies

Another important concept in ShapeShifter is that of a node. A node is a named object in a hierarchy of other objects. A node has a parent and may have one or more children. Each node has a name that is unique within the scope of its parent. The fully qualified name of a node includes the names of the node’s parents separated by the ‘.’ delimiter. In other words, it’s a named object hierarchy.

Objects that are nodes implement the INode interface. Nodes that may contain children implement the ICompositeNode interface, which is derived from INode. Shapes, meshes, textures, materials, and lights are examples of nodes. Models are also nodes, which means that they can be nested inside of other models. 3.3        Visual Objects

A visual object is any type of node that can be rendered and that has three-dimensional bounds in world space. Visual objects implement the IVisual interface, which derives from the ICompositeNode interface. Visual objects have a bounding box that defines the bounds of the object in world coordinates. Transformations such as translation (move), rotation, and scaling can be applied to visual objects.

Because visuals are composite nodes, they may be composed of other visual objects. Complex hierarchies of visuals can be created through composition. Translation, rotation, and scaling are always performed relative to the transformations of the parent visual objects. The object hierarchy is traversed in a depth-first manner and the transformation matrix of each node is pushed onto the transform stack. This assures that transformations in parent nodes are combined with a visual’s own transformations in order to produce a resulting transformation matrix that orients the visual object with respect to its parents.

3.4        Meshes

A mesh is a type of node that contains vertices and rendering primitives that define a wire frame. A mesh may contain one or more rendering primitives. Point lists, line lists, line strips, triangle lists, triangle fans, triangle strips are the types of primitives that may be in a mesh. Each primitive contains a list of vertices, which are actually index values into the mesh’s vertex buffer. The vertex buffer is an array containing all of the vertices used by the primitives in the mesh. Indexed primitives are used in order to reduce the total number of vertices required to render the mesh, since entries in the vertex buffer can be shared by multiple primitives.

3.5        Shapes

A shape is a type of visual node that contains a mesh and the context information for rendering the mesh. Context information for the mesh may consist of textures, materials, and rendering state flags. A primitive is the smallest rendering unit to which rendering context information can be applied. The shape provides allows context objects to be associated with each primitive in the mesh. Multiple shapes can share the same mesh and apply different context objects to the mesh. Like other visual objects, shapes support composition and may contain child nodes.

3.6        Device Resources

In Direct3D, objects such as textures, vertex buffers, and index buffers are always tied to a given device (i.e. IDirect3DDevice8).  ShapeShifter decouples nodes in the model from a particular device by encapsulating device dependent resources like textures, vertex buffers, and index buffers in a ShapeShifter Device object. Rather than storing direct pointers from nodes in the model to these device dependent resources, resource identifiers are stored instead which are used to retrieve the resources from the ShapeShifter device when they are needed. This keeps the model free of device dependencies and consolidates ownership and management of device resources in the ShapeShifter Device class.

For example, a texture node in the model might be assigned a resource ID of 21. When the texture node is asked to select its texture into a Direct3D device, it passes its resource ID to the Device and gets back the actual IDirect3DTexture8 pointer. This lookup is very quick because the resource ID is just an index into an array and the overhead is one v-table call and an array access.

The method LoadDevice defined in the IDeviceResource interface provides a mechanism for nodes to create and load their resources onto the device. Nodes that require device resources implement this interface. LoadDevice must be called once before a node can participate in the rendering process and must be called anytime the contents of the node changes. For example, after loading a mesh the application must call LoadDevice for each device it will be rendering to (typically an application uses only one device).  From that point, the mesh can be rendered over and over without calling LoadDevice again. If a change is made to the mesh, such as moving or adding vertices, then LoadDevice must be called in order to update the device resource. The IDeviceResource interface provides a corresponding UnloadDevice method to allow a node to release any device resources it is using.

3.7        Context Objects

Context objects are nodes that provide state information and resources to the device during rendering. Materials, textures, texture stages, and lights are examples of context objects. Context objects prepare the device for rendering one or more primitives. All context objects implement the IContextObject interface, which is a custom interface (see the section called C++ Only Interfaces) that defines a method for selecting the object into a Direct3D device (IDirect3DDevice8).

The RenderState class is a special context object that aggregates other context objects and maintains a list of rendering states. A rendering state is an ID/value pair that sends an instruction to the Direct3D device to control features such as Z-buffering, fill mode, culling mode, ambient light, alpha blending, anti-aliasing, and fog. The ID/value pair is passed to the Direct3D device through the IDirect3DDevice8::SetRenderState method. Any rendering state that can be passed to this function can be assigned to a RenderState object. The Direct3D enumeration D3DRENDERSTATETYPE defines the valid IDs that be assigned to a RenderState object. The value associated with the ID is always a DWORD.

The RenderState class may also contain other context objects such as textures, materials, texture stages, and lights. This is especially useful for bundling a texture and material together with one or more rendering states. The resulting RenderState object can be treated like any other context object. Context objects are added to a RenderState object using the AppendChild and InsertChild methods defined by the ICompositeNode interface.

The Shape class provides an interface for associating context objects with each primitive in the mesh. Each primitive can be associated with one context object. You can associate any kind of context object with a primitive such as a texture, material, or render state object. If you need to associate more than one context object or you need to associate rendering states with a primitive, use a RenderState object that aggregates the other context objects.

3.8        C++ Only Interfaces

ShapeShifter defines several interfaces that are not automation compatible and can only be used by C++ clients and the ShapeShifter library. For an interface to be automation compatible (i.e. usable by scripting clients like VBScript), all parameters must be VARIANTs or VARIANT compatible types. Interfaces that are not automation compatible are also referred to as custom COM interfaces.

The purpose of these custom interfaces is to allow DirectX structures and interfaces and custom data types to be passed as parameters. Custom interfaces that pass DirectX 8 data types as parameters are given the suffix DX8. For example, the IRenderDX8 interface has a method that takes a pointer to an IDirect3DDevice8 interface. Some of these custom interfaces extend the automation compatible, dual interfaces that are defined in the type library. For example, the IMeshEx interface extends the IMesh interface with methods that take vertex arrays as parameters in order to provide fast loading and initialization of meshes. Custom interfaces that extend the automation-compatible interfaces are given the suffix Ex.

The custom interfaces are listed below with a brief description of each.

IRenderDX8 – Defines a Render method that renders the object to a Direct3D device.

IContextObject – Defines a SelectObject method which prepares a Direct3D device for rendering by selecting resources and render states.

IContextProvider – Defines a SelectContext method which selects one or more context objects into a Direct3D device given a context identifier. This interface is implemented by shape classes and is called by the mesh during rendering in order to prepare the device for rendering a given primitive. In that scenario, the context identifier is passed to SelectContext is the primitive index.

IMeshEx – Extends the IMesh interface with methods for loading the mesh given raw vertex arrays. Speeds up loading of meshes by allowing entire blocks of vertices and primitives to be loaded in a single call. Vertex arrays are passed as VOID* parameters to allow Direct3D flexible vertex formats to be used by the mesh.

ICameraEx – Extends the ICamera interface with methods to get and set the camera location and orientation in a single function call. Also provides a method for calculating the view matrix.

IDrawMeshDX8 – Implemented by mesh classes, this interface provides methods to render the entire mesh or individual primitives within the mesh.

IResourceManagerDX8 – This interface is implemented by the device class and provides methods for accessing resources such as textures and vertex buffers that are managed by the device. Objects use this interface during rendering in order to lookup a resource ID and get back the actual resource.

IInitInputDeviceDX8 – Defines the _InitObject method used to initialize an input device. The IDirectInput8 interface and DIDEVICEINSTANCE structure are passed as parameters.

IInputDeviceProviderDX8 – Defines the GetInputDevice method which is used to retrieve an input device based on a given input device category. Controllers implement this interface and use it to expose input devices that are attached to it.

IInputDeviceUserDX8 – Implemented by objects that use the IInputDeviceProviderDX8 interface. Defines two methods: AcquireInputDevices and FreeInputDevices which are hooks that allow the object acquire and release the input devices it needs. This interface is implemented by behavior objects.

4          How To

This section explains how to accomplish specific tasks in ShapeShifter.

4.1        Initialize ShapeShifter

The first step in initializing ShapeShifter is to create an engine object. The Engine class manages the available rendering and input devices and takes cares of initializing DirectX. If you are using the ShapeShifterCtl ActiveX control, this step is taken care of for you automatically by the control when it is created.

The sample code below shows how to create and initialize the engine in C++. First, declare the engine in your CWinApp derived class.

class CMyApp : public CWinApp

{

public:

   CMyApp();

 

   IEnginePtr m_spEngine;

...

 

Next, create the engine object in the InitInstance function and initialize it.

if (SUCCEEDED(m_spEngine.CreateInstance(__uuidof(Engine))))

{

   m_spEngine->InitializeEngine();

}

 

The same code in Visual Basic looks like this.

Dim ShapeShifterEngine As ShapeShifter.Engine

 

Private Sub Form_Load()

    Set ShapeShifterEngine = New ShapeShifter.Engine

    ShapeShifterEngine.InitializeEngine

End Sub

 

4.2        Select a device

Once the engine has been created and initialized, you must select a device on which to render your viewports. Typically, there is only one device on a system unless you have more than one video adapter installed. You can either have the engine give you the default device for the system or you can iterate through the available devices and select the most appropriate device.

The following Visual Basic code shows how to iterate through the available devices.

Dim CurDevice As ShapeShifter.Device

Dim DevIdx As Integer

For DevIdx = 0 To ShapeShifterEngine.GetDeviceCount - 1

Set CurDevice = ShapeShifterEngine.GetDevice(DevIdx)

Next DevIdx

 

The following C++ code shows how to select the default device.

m_spDevice = m_spEngine->GetDefaultDevice();

4.3        Load a model from a file

ShapeShifter encapsulates the responsibility for reading and writing the various supported graphics file formats into reader and writer objects. For each supported file format there is a reader class that implements the IModelReader interface and possibly a writer class that implements the IModelWriter interface. Loading a model from a file entails creating an instance of reader class for the given file format and calling the Read method. The Read method takes the filename and a pointer to the model as parameters.

The following Visual Basic code shows an example of loading a 3DS (3D Studio Max) file.

Dim Model As ShapeShifter.Model

Dim Reader As ShapeShifter.ModelReader3DS

Set Model = new ShapeShifter.Model

Set Reader = new ShapeShifter.ModelReader3DS

Reader.Read “mymodel.3ds”, Model

 

If you are using the ShapeShifterCtl ActiveX control, things are even simpler. All you have to do is assign the filename to the FileName property and call the LoadFile method. The control automatically detects the file type and creates the appropriate type of reader object. The following Visual Basic code shows an example of loading a 3DS file through the ActiveX control.

ShapeShifterCtl1.ModelFile = “mymodel.3ds”

ShapeShifterCtl1.LoadFile

 

4.4        Load a model programmatically

It’s quite possible that you will want to create objects at runtime and add them to your model. Your model may be a mix of objects loaded from external files and objects created dynamically at runtime or it may be entirely created at runtime. New nodes can be created in the model using the CreateNode method. You can use it to create shapes, meshes, textures, materials, and any other kind of node in the model.

The Visual Basic example below shows how to load a new shape into the model, edit its mesh, and assign it a material.

 

Public Sub LoadModel(Dim Model As ShapeShifter.Model)

   Dim MyShape As ShapeShifter.CompositeShape

   Dim MshEdtr As ShapeShifter.MeshEditor

   Dim MyMaterial As ShapeShifter.Material

 

   ' Create a new shape at the top level of the model. Its fully

   ' qualified name is <ModelName>.MyShape

 

   Set MyShape = Model.CreateNode("MyShape",

                                  "ShapeShifter.CompositeShape",

                                  "")

 

   ' Use the mesh editor to edit the shape’s mesh. The mesh is

   ' created implicitly by the shape when the Mesh property is

   ' first referenced. It is created as a child node of the

   ' shape and its fully qualified name is

' <ModelName>.MyShape.Mesh.

     

Set MshEdtr = New ShapeShifter.MeshEditor

MshEdtr.BeginEdit MyShape.Mesh

MshEdtr.MakeSphere 300, 16, 16

MshEdtr.EndEdit

 

   ' Create a new material at the top level of the model. Assign

   ' the material to the shape and associate it with the first

   ' (and only in this case) primitive in the mesh.

 

   Set MyMaterial = Model.CreateNode("MyMaterial",

                                     "ShapeShifter.Material ",

                                     "")

   MyShape.SetContext 0, MyMaterial

End Sub

 

4.5        Organize nodes in a hierarchy

ShapeShifter provides a very flexible way to organize the nodes in a model. You can nest nodes inside other nodes and create a well-organized hierarchy of named objects. You can nest shapes any number of levels deep and even nest models inside of other models. You can also organize meshes, materials, textures, texture stages, rendering contexts, and lights inside other nodes. The IModel::CreateNode method provides the primary means of organizing nodes in a hierarchy. 4.6        Create a viewport

A viewport is a two-dimensional window onto a device. Viewports are created through a Device object and are associated with that device for their entire lifetime. There are two kinds of viewports: windowed and full-screen. Windowed viewports exist inside of a window and are clipped to the window. A full-screen viewport takes up the entire screen and only one can be active at a time.

The Visual Basic code below shows how to create and initialize a windowed viewport.

 

Public Function CreateViewport(Device As ShapeShifter.Device)

       As ShapeShifter.Viewport

 

   ' Create the viewport

 

   Dim Viewport As ShapeShifter.Viewport

   Set Viewport =

            Device.CreateViewport("ShapeShifter.WindowedViewport")

   

   ' Set the viewport bounds so that it occupies

   ' the entire client area of the form

 

   Viewport.Left = Main.ScaleLeft

   Viewport.Top = Main.ScaleTop

   Viewport.Width = Main.ScaleWidth

   Viewport.Height = Main.ScaleHeight

   

   ' Assign the window handle of the form to the viewport

 

   Viewport.ContainerWindow = hWnd

   

  ' Initialize the camera so that is on the Z axis looking

  ' straight into the scene

 

   Viewport.Camera.SetLocation 0, 0, -8

   Viewport.Camera.SetAimVector 0, 0, 1

   

   ' Set the viewing volume

 

   Viewport.SetFrustum 1.57, 1, 1, 1000

   

   ' Enable Z buffering

 

   Viewport.ZEnable = 1

 

   Set CreateViewport = Viewport

End Function

 

If you are using the ShapeShifterCtl ActiveX control, creation of the viewport is done automatically. The ViewportType property determines the type of viewport created. You can access ShapeShifterCtl’s viewport through the Viewport property.

4.10    Add lighting

Lights come in three flavors: directional lights, spotlights, and point lights. A light is a context node that implements the ILight interface.

Directional lights cast light in a single direction throughout the entire scene. It can be thought of as parallel rays of light that travel for what is effectively an infinite distance. Directional lights have not point of origin and do not attenuate over distance. This type of light is the least expensive in terms of rendering overhead and is useful for modeling a light source like the sun. The DirectionalLight class implements directional lights in ShapeShifter.

Point lights give off light in all directions equally from a particular point. Point lights are affected by attenuation (i.e. their intensity varies over distance). The distance of a point light from a given vertex will determine how much light the vertex receives. A point light is similar to a light bulb. Point lights are implemented by the PointLight class.

Spotlights have a point of origin and a direction. They are similar to point lights but only emit light within a conical region of world space. A desk lamp or a car’s headlights are examples of spotlights. The ISpotLight interface extends ILight and defines methods for configuring a spotlight. The SpotLight class provides the implementation for spotlights.

The following Visual Basic sample shows how to create a directional light in the model.

 

Public Function CreateDirLight() As ShapeShifter.DirectionalLight

  

   ' Create a directional light in the model. The Model is

   ' assumed to be a form-level variable.

 

   Dim DirLight As ShapeShifter.DirectionalLight

   Set DirLight = Model.CreateNode("LetThereBeLight",

                                "ShapeShifter.DirectionalLight",

                                "")

 

 

   ' Set the direction vector of the light to point straight

   ' down the Y-axis (directly overhead)

 

   DirLight.SetDirection 0, -1, 0

 

   ' Set the ambient and diffuse lighting characteristics. No

   ' specular value is assigned so this light will not produce

   ' specular light.

 

   DirLight.SetAmbientColor 0.5, 0.5, 0.5

   DirLight.SetDiffuseColor 0.5, 0.5, 0.5

 

   Set CreateDirLight = DirLight

 

End Function

Creation of point lights and spotlights is identical to the example above, except for having to set a few additional properties on the light. Point lights and spotlights require that the position in world space be supplied. Both point lights and spotlights are affected by attenuation, which can be controlled through several properties defined by the ILight interface. Note that setting attenuation properties for a directional light has no affect since directional lights do not change intensity over distance. Finally, spotlights require that a direction and angle for the light cone be supplied.

Ambient light affects an entire scene and has no particular point of origin or directional. Ambient light is set at the model level through the IModel interface. Only materials that have ambient light values are illuminated by ambient light.

4.11    Apply textures

A texture is a bitmap image that is applied to one or more primitives during rendering. Think of a texture as a bitmap of pixel colors that you apply to the surface of a 3D object. Textures provide added detail to a 3D object and make the scene look more realistic.

Textures are stored in the model as nodes that implement the ITexture interface. They are typically loaded from a bitmap file stored in one of the following formats: BMP, TGA, PNG, JPG, DIB, PPM, and DDS. DDS files are DirectDraw surface files and can be created using the DXTex utility that ships with DirectX. To load a texture from a file just set the FileName property of the texture object.

Textures are context objects and therefore implement the IContextObject interface. The texture node in the model stores a device resource ID that references the actual texture on the device. During rendering, the texture is selected into the device using the device resource ID. Prior to rendering, the texture must have been loaded onto the device with a call to LoadDevice.  The LoadDevice function reads the texture from the external file and creates a Direct3D texture (i.e. IDirect3DTexture8) on the device and associates it with the resource ID. If a resource ID was not previously assigned to the texture node, a unique ID is generated during the call to LoadDevice.

Textures are mapped onto a primitive on a vertex-by-vertex basis. Texture coordinates stored in each vertex determine which part of the texture is mapped onto the primitive. Each texture coordinate is a pair of values (u,v) that specify a percentage along the width and height of the two-dimensional bitmap. The texture coordinate (0.0, 0.0) refers to the upper-left hand corner of the bitmap and (1.0, 1.0) refers to the lower-left hand corner of the bitmap. The texture coordinate (0.5, 0.5) refers to the center of the bitmap. The MeshEditor class can be used to set the texture coordinates of vertices in a mesh. The following Visual Basic code sets the texture coordinates for cube mesh that will have a texture applied to each of its six sides.

 

Private Sub SetCubeTextureCoordinates(Mesh As ShapeShifter.Mesh)

 

   Dim MeshEditor As ShapeShifter.MeshEditor

   Set MeshEditor = new ShapeShifter.MeshEditor

      MeshEditor.BeginEdit Mesh

 

   MeshEditor.UpdateTextureCoordinate 0, 0, 1

   MeshEditor.UpdateTextureCoordinate 1, 0, 0

   MeshEditor.UpdateTextureCoordinate 2, 1, 1

   MeshEditor.UpdateTextureCoordinate 3, 1, 1

   MeshEditor.UpdateTextureCoordinate 4, 0, 0

   MeshEditor.UpdateTextureCoordinate 5, 1, 0

 

   MeshEditor.UpdateTextureCoordinate 6, 0, 1

   MeshEditor.UpdateTextureCoordinate 7, 0, 0

   MeshEditor.UpdateTextureCoordinate 8, 1, 1

   MeshEditor.UpdateTextureCoordinate 9, 1, 1

   MeshEditor.UpdateTextureCoordinate 10, 0, 0

   MeshEditor.UpdateTextureCoordinate 11, 1, 0

 

   MeshEditor.UpdateTextureCoordinate 12, 0, 1

   MeshEditor.UpdateTextureCoordinate 13, 0, 0

   MeshEditor.UpdateTextureCoordinate 14, 1, 1

   MeshEditor.UpdateTextureCoordinate 15, 1, 1

   MeshEditor.UpdateTextureCoordinate 16, 0, 0

   MeshEditor.UpdateTextureCoordinate 17, 1, 0

  

   MeshEditor.UpdateTextureCoordinate 18, 0, 1

   MeshEditor.UpdateTextureCoordinate 19, 0, 0

   MeshEditor.UpdateTextureCoordinate 20, 1, 1

   MeshEditor.UpdateTextureCoordinate 21, 1, 1

   MeshEditor.UpdateTextureCoordinate 22, 0, 0

   MeshEditor.UpdateTextureCoordinate 23, 1, 0

 

   MeshEditor.EndEdit

End Sub

 

The sample above assumes that the mesh contains a single triangle list that forms a cube. The first parameter passed to UpdateTextureCoordinate identifies the vertex that will be updated with the texture coordinate. The second and third parameters are the (u,v) texture coordinates.

In order to apply a texture to a primitive, it must be associated with a shape with a call to IShape::SetContext. The first parameter to SetContext identifies the primitive in the shape’s mesh that the texture will be applied to. The following Visual Basic code creates a texture node and associates it with the first primitive in a given shape.

 

Public Function CreateTexture(Shape As ShapeShifter.Shape,

                              FileName As String) 

       As ShapeShifter.Texture

 

   ' Create the new texture. Model is assumed to be a

   ' form-level variable

 

   Dim Texture As ShapeShifter.Texture

      Set Texture = Model.CreateNode("MyTexture",

                               "ShapeShifter.Texture ",

                               "")

 

   ' Assign the filename

 

   Texture.FileName = "mytexture.bmp"

 

   ' Associate the texture with primitive 0 (i.e. first primitive)

 

   Shape.SetContext 0, Texture

 

   ' Return the texture to the caller

 

 

   Set CreateTexture = Texture

 

End Function

 

Associating a texture with a primitive is usually not sufficient to get the results you want, unless you want to use the default texture stage states when the texture is mapped. Texture stage states are instructions to the Direct3D device that control various texture mapping options such as the addressing mode, magnification filtering, minification filtering, and alpha blending. A texture stage state consists two DWORDs: a type identifier and a value. The D3DTEXTURESTAGESTATETYPE enumeration provided by Direct3D defines the valid types of texture stage states. For a complete discussion of texture stage states, refer to the DirectX SDK Documentation.

ShapeShifter provides a TextureStage class that encapsulates a texture and a list of texture stage states that are applied when the texture is used. The TextureStage class implements the ITextureStage interface, which exposes the texture as a property and provides methods for adding texture stage states. The following Visual Basic code creates a texture stage and associates it with the first primitive in a given shape.

 

Public Function CreateTextureStage(Shape As ShapeShifter.Shape,

                                   FileName As String) 

       As ShapeShifter.TextureStage

 

   ' Create the new texture stage. Model is assumed to be

   ' a form-level variable

 

   Dim TextureStage As ShapeShifter.TextureStage

   Set TextureStage = Model.CreateNode("MyTextureStage",

                                    "ShapeShifter.TextureStage",

                                    "")

 

   ' Assign the filename

 

TextureStage.Texture.FileName = "mytexture.bmp"

 

   ' Set the magnification filter to linear

 

   TextureStage.MagFilter = SSCLTEXF_LINEAR

 

   ' Associate the texture stage with primitive 0

' (i.e. first primitive)

 

   Shape.SetContext 0, TextureStage

 

   ' Return the texture to the caller

 

   Set CreateTextureStage = TextureStage

 

End Function

 

Create a texture stage is just as easy as creating a texture and it provides more control over the texture mapping process. The TextureStage class defines properties for some of the more commonly used texture stage states. Instead of using the MagFilter property we could just as easily use the SetTextureState function. The first parameter to the SetTextureState function is the state type, and the values of this parameter map directly to the D3DTEXTURESTAGESTATETYPE enumeration defined by Direct3D.

Another important function of the TextureStage class is multi-texturing. Many interesting special effects can be created by blending textures in multiple stages. Each stage has a texture and a set of states that determine how the texture is combined with previous stages. Refer to the DirectX SDK Documentation for a complete discussion of multi-texturing.

In order to perform multi-texturing in ShapeShifter, a RenderingContext object can be used to aggregate the texture stages into a single context object. The TextureStage class has a StageNum property that must be assigned in order to specify the order in which the texture stages are applied.

Typically, you will want to tie a texture and material together into a single context object. This is another good reason to put your textures into a RenderingContext object. That way, you can combine with a material and any rendering states you may need to set.

4.12    Execute commands

A command is an object that encapsulates an action on a model. Commands encapsulate the action and the parameters used to perform the action. You can execute and undo commands through the controller. The IController interface is derived from the ICommandDispatcher interface, which defines methods for executing and undoing commands. The controller and its behaviors generate commands in response to user input. You can also send commands directly to the controller from your own code.

Here is a list of command classes currently defined by ShapeShifter. This list will grow in future releases to include new commands.

InsertCommand – Inserts one or more nodes into a model

DeleteCommand – Removes one or more nodes from a model

MoveCommand – Translates one or more visual nodes by a given delta vector

RotateCommand – Rotates one or more visual nodes by a given angle (specified in radians) about a vector in world space

ScaleCommand – Scales (resizes) one or more visual nodes by a given percentage

MacroCommand – Aggregates one or more other commands into a single command

In order to execute a command, you must create a command object that is an instance of one the classes listed above. Then call the ICommandDispatcher::Do method on your controller object, passing the command you have created. After the command is executed successfully, the controller pushes a reference to your command object onto an undo stack. You can undo one or more commands by calling the ICommandDispatcher::Undo method on your controller object, passing the number of commands to undo. The controller pops the undo stack the appropriate number of times calls each command’s Undo method.

The following Visual Basic code shows how to execute a move command.

Pubic Sub MoveShape(ShapeName As String)

 

   Dim Shape As ShapeShifter.Node

   Dim Cmd As ShapeShifter.MoveCommand

 

   ' Get the existing shape from the model. Model is

   ' assumed to be a form-level variable.

 

   Set Shape = Model.GetChildByName(ShapeName)

 

   ' Create a move command and add the shape as a parameter

 

   Set Cmd = new ShapeShifter.MoveCommand

   Cmd.AddNode Shape

 

   ' Tell the command how far to move the shape

   ' each direction

 

   Cmd.SetDelta 4, 5, 6

 

   ' Pass the command to the controller for execution. Controller

   ' is assumed to be a form-level variable.

 

   Controller.Do Cmd

 

End Sub
